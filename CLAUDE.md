# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build & Run

Prerequisite: `Config/Secrets.xcconfig` must exist (see next section) — both build and tests fail without it. Tests run hosted in the app, so an empty secret value crashes the test run with `fatalError` rather than a normal assertion failure.

```bash
# Build (simulator)
xcodebuild -project kolco24.xcodeproj -scheme kolco24 -destination 'platform=iOS Simulator,name=iPhone 16' build

# Run unit tests
xcodebuild test -project kolco24.xcodeproj -scheme kolco24 -destination 'platform=iOS Simulator,name=iPhone 16'

# Run a single test class
xcodebuild test -project kolco24.xcodeproj -scheme kolco24 -destination 'platform=iOS Simulator,name=iPhone 16' -only-testing:kolco24Tests/SecretsTests
```

If the name-based destination is flaky, resolve a UDID via `xcrun simctl list devices available` and use `-destination 'platform=iOS Simulator,id=<UDID>'` (may also need `DEVELOPER_DIR=/Applications/Xcode.app/Contents/Developer`).

Open `kolco24.xcodeproj` in Xcode to run on a simulator interactively.

## Building from Scratch (Secrets)

A fresh clone does not build until secrets are in place:

```bash
cp Config/Secrets.example.xcconfig Config/Secrets.xcconfig
# then fill in the real values
```

Without it the build fails with `could not find included file 'Secrets.xcconfig'` — a deliberate hard gate: the committed `Config/App.xcconfig` (base configuration of the app target) does a non-optional `#include "Secrets.xcconfig"`, and `Config/Secrets.xcconfig` is gitignored. Values flow xcconfig → `kolco24/Info.plist` (`Kolco24*` keys, merged with the generated plist) → `enum Secrets` in `kolco24/Secrets.swift` (`apiBaseURL`, `appKeyId`, `appSecret`, `localAPIBaseURL`; missing/empty value → `fatalError`). Gotcha: `//` starts a comment in xcconfig, so URLs use the `$()` trick (`https:/$()/...`).

**ATS:** `Info.plist` sets `NSAppTransportSecurity` → `NSAllowsLocalNetworking = YES` — cleartext HTTP is allowed only within the local network (any LAN/.local host, deliberately broader than Android's per-IP pin: `NSExceptionDomains` can't take IP addresses); the cloud API stays HTTPS-only.

**Dependencies:** GRDB.swift 7.x via SPM (SQLite, Room analog), linked to the app target only — tests import it through the host application.

**Project file gotcha:** `kolco24/` is a synchronized group (`PBXFileSystemSynchronizedRootGroup` — files dropped there auto-join the target), so `kolco24/Info.plist` is excluded via a `PBXFileSystemSynchronizedBuildFileExceptionSet` with `membershipExceptions = (Info.plist)` to keep it from being copied into the bundle as a resource; any file under `kolco24/` needing special target membership needs the same treatment. `GENERATE_INFOPLIST_FILE = YES` stays on — Xcode merges the partial plist with the generated keys (locked in by `InfoPlistTests`).

## Architecture

SwiftUI iOS app for a rogaine/orienteering event. UI language is Russian. The UI/app files (`kolco24App`, `ContentView`, `MarksView`, `LegendView`, `TeamView`, `ScanSheet`, `SharedComponents`, `DesignTokens`, plus the stage-4 selection-flow views `CompPickerView`, `TeamPickerView`, `TeamConfirmSheet`, `EmptyStates`, and the stage-5 `BindChipSheet`) plus `Secrets` live flat in the root of `kolco24/`; the `@Observable` model layer lives in `kolco24/App/` (see "App layer") and the pure-logic port in `kolco24/Core/` and `kolco24/Model/` (see "Pure logic layer" below).

**Entry point:** `kolco24App.swift` → `ContentView.swift` (3-tab `TabView`, tinted `Color.kolcoOrange`, haptic on tab switch)

**Tabs:**
- `MarksView` — 4-column grid of taken checkpoints (КП), metrics card, floating NFC/Photo CTA, scan sheet modal
- `LegendView` — filterable list of all checkpoints with score progress strip (`.insetGrouped` List)
- `TeamView` — team hero card, member list with chip-binding status, misc settings rows

**Design system (`DesignTokens.swift`):**
- Color palette "A2 Grey v2", **adaptive (light/dark)**: `Color.ink`, `Color.sub`, `Color.paper`, `Color.brandRed`, `Color.kolcoOrange`, `Color.good`, `Color.charcoal`, `Color.charcoalHi`. `Color.amber` is the only non-adaptive token (same in both themes via `Color(hex:)`).
- Surface/line/shadow tokens (replace former literals like `Color.white` / `Color.black.opacity(...)` in views): `Color.card`, `Color.hairline`, `Color.cardShadow`.
- Adaptive tokens are built with `Color(light:dark:)` (two hex strings) or `Color(lightUI:darkUI:)` (two `UIColor`s, for tokens carrying their own alpha — `hairline`, `cardShadow`). Both wrap `UIColor(dynamicProvider:)`, so the system switches the whole palette by `userInterfaceStyle` trait — there is no theme toggle, screen, or persistence. Dark hex values mirror the token table in `tmp/design_dark.html`.
- Typography — `Font.mono(_:weight:)` wraps JetBrains Mono (must be bundled + declared in `Info.plist` under `UIAppFonts`)
- Spacing/radius constants in `enum DS` — `DS.hPad` (16), `DS.cardRadius` (13), `DS.heroRadius` (18), `DS.ctaRadius` (16)

**Shared components (`SharedComponents.swift`):**
- `CPBadge` — checkpoint number badge with red top/bottom stripes; `number: "?"` renders faded
- `MetricView` — labelled metric with optional unit; `isWarning: true` renders value in `brandRed` with mono font
- `VDivider`, `SectionHeader`, `GreenCheckCircle`
- `DarkHeroBackground` — charcoal gradient + radial red accent + diagonal `Canvas` line pattern; used by `TeamHeroView` (in `TeamView.swift`) and `TimerHeroView` (in `ScanSheet.swift`)

**Data model pattern:** The 3 tabs render from **real DB data** (stage 4 wired the UI to the persistence/sync slab via the `App/` model layer — see "App layer" below); the mock arrays are gone. The old view-owned mock structs `CheckpointTile`/`LegendCP`/`TeamMember` were removed — views now render straight from domain types (`Checkpoint`, `Team.TeamMemberItem`) and Core display types (e.g. `MarkTile` from `Core/Marks/MarksDisplay`), built by the pure mappers in `Core/`. (Stage 5 wired `ScanSheet` and the NFC scan/bind flow to real data — see "Scan & bind layer" below; the old mock chip types are gone.)

**NFC scan flow:** `FloatingCTAView` in `MarksView` presents `ScanSheet(model:)` via `.sheet(item:)`, driven by a live `ScanModel` (stage 5 — see "Scan & bind layer"). `ScanSheet` renders `CPCardView` (scanned КП badge / number+cost / done state), a 2-column `ChipSlotView` grid (present + buffered slots), a `DiagnosticLine`, and `TimerHeroView` (countdown circle + remaining scans) — all from real model state, no stubs.

**Recurring visual motif:** Diagonal `Canvas` line patterns appear in `NFCTileView`, `PhotoTileView`, and `DarkHeroBackground` — use the same approach when adding new tile/card types.

**Fixed-dark surfaces:** `DarkHeroBackground` (and its `TeamHeroView` / `TimerHeroView` users) and `NFCTileView` stay dark in *both* themes, so their white content reads correctly either way — but by different means. `DarkHeroBackground`'s gradient is built from the **adaptive** `charcoal`/`charcoalHi` tokens (both dark-valued in either theme, so it mirrors each design: `1D242D → 2A323C` light, `27313D → 171D25` dark); only its text/hatch content is theme-independent. `NFCTileView` is fully fixed via literal `Color(hex:)`, **not** adaptive tokens — a "chip card": a `#171D25 → #232A33` gradient with inset depth, a faint diagonal `Canvas` hatch, a three-arc contactless glyph (`#E6EAF0`), and a white mono number — no red reflective stripes (those belong to the light `CPBadge`, which is unchanged).

**Removed:** the `isRecent` field/feature (green ring overlay on tiles) no longer exists — don't reintroduce it.

**`LegendCP.display`** formats as `"{cost}-{number}"` (e.g. `"4-07"`) for the legend list identifier column.

- **`kolco24/Model/`** — domain value types mirroring Room v5 (no `Entity` suffix; `Model/` itself stays free of `import GRDB` — the GRDB `FetchableRecord`/`PersistableRecord` conformances landed in stage 2 as `Data/Records/<Type>+GRDB.swift` extensions, keeping the value types Android-/GRDB-free). Stage 1 added `Checkpoint`, `Mark` (+ nested `MarkMemberSnapshot`), `MemberChipBinding`, and the `UnlockOutcome` enum (`revealed`/`identityOnly`/`unknown`/`failed`); stage 2 added the 10 remaining Room-v5 domain types: `Race`, `Category`, `Team` (+ nested `TeamMemberItem`), `SelectedTeam`, `Tag`, `MemberTag`, `LegendMeta`, `TrackPoint`, `JudgeScan`, `SyncMeta`. Kotlin sealed hierarchies became Swift `enum`s with associated values; data classes became `Equatable` structs; `ByteArray` → `Data`/`[UInt8]`; `Long` → `Int64`, `Float` → `Float`.

**`TrustedClock` (actor).** Ports `data/time/TrustedClock.kt` behavior 1:1 (server anchor pinned to a monotonic `elapsedRealtime`-style read, boot-session identity, monotonic-regression reboot detection, skew) but swaps the concurrency idiom: an `actor` (isolation serializes all reads/writes; callers `await`) replaces Kotlin's `AtomicReference` + `synchronized`. Instead of `StateFlow` it exposes an isolated `status: ClockStatus` (`noSync`/`ok`/`skewed(skewMs:)`, `SKEW_THRESHOLD_MS = 60_000`) plus a `nonisolated statusUpdates: AsyncStream<ClockStatus>` (`.bufferingNewest(1)`, equal values deduped by hand — the stage-11 skew banner is the consumer). Time sources are injected closures (`elapsedProvider`, `wallProvider`, `bootCountProvider`, `persist`/`persisted`), so the core is Android-/UI-free and async-unit-testable with fakes. Production providers live in `SystemClockProviders.swift`: elapsed = `mach_continuous_time()` in ms (keeps running while the device sleeps, unlike `systemUptime`/`CLOCK_UPTIME_RAW`), wall = `Date().timeIntervalSince1970 * 1000`, and `bootCount` is always `nil` (no iOS analog of `Settings.Global.BOOT_COUNT` — the ported logic treats `nil` as "no reboot evidence" and catches reboots via monotonic-clock regression against the saved anchor).

**KAT (known-answer test) vectors — byte-for-byte server interop.**
- `LegendCrypto`: `LegendCryptoTests` embeds a server-generated vector (`bid`, `wrapKey`, `iv`/`ct`, `aad`, plaintext) produced by a throwaway scratchpad script over the server reference `crypto.py`/`legend_crypto.py` (`src/apps/mobile/`) — importing `seal`/`derive_wrap_key`, reproducing the Django-ORM-bound `bid` and bundle-map one-liners. This proves HKDF (`salt=None` → 32 zero bytes), `bid`, and AAD interop. (The 4 corresponding Android tests are `@Ignore`d with `TODO(server-vector)`.)
- HMAC: the signing KAT is inline in `SigningTests` — `sign("test-secret-123", <canonical>)` = `cf1c254fb2eac6c7efde1cff6efe9553878370299cd60a42be4d2105a8072588`, cross-checked against a Python `hmac` one-liner.

**Main port trap — hex from signed bytes.** Kotlin smears `"%02x".format(byte)` / `b.toInt() and 0xFF` across files; `%02x` on a negative `Int8` in Swift sign-extends and corrupts the output. Everything therefore runs on `UInt8`/`Data`, and hex encode/decode goes through the single `Core/Util/HexBytes.swift` helper (nibble-indexed digit table — `v >> 4` / `v & 0x0F`) — it centralizes the hex logic Kotlin spreads across files as `%02x` / `b.toInt() and 0xFF` (the raw `& 0xFF` byte normalizations live in `ChipRecord.swift`).

## Data layer (`Data/`)

Android-port stage 2 (see [`docs/plans/completed/20260709-android-port-stage2.md`](docs/plans/completed/20260709-android-port-stage2.md)) added the GRDB persistence slab, ported 1:1 from `kolco24_app_v2`'s Room v5 (`data/db/`) with its DAO/converter tests mirrored into `kolco24Tests/Data/`. `import GRDB` lives **only** under `Data/`; `Core/`/`Model/` stay GRDB-free (grep-invariant, extending the stage-1 no-`UIKit`/`SwiftUI` rule — which also holds for `Data/`).

- **`kolco24/Data/AppDatabase.swift`** — wraps `any DatabaseWriter` + a `DatabaseMigrator` with a **single** migration `"v1"` = the finished Room v5 schema snapshot transcribed verbatim from `app/schemas/…/5.json` (the 1→5 migration history is *not* replayed — the iOS DB is born at v5). All 13 tables, 4 composite PKs, all indexes, **no SQL `DEFAULT`** (there are none in `5.json`'s `createSql` — Room defaults are Kotlin-side, applied in Swift initializers instead). No FKs anywhere (matches Room; do not add — it would change `replaceAllForRace` behavior). `makeShared()` = `DatabasePool` (WAL) at `kolco24.db` in Application Support; `makeInMemory()` = `DatabaseQueue()` for tests. Schema inventory is locked by `AppDatabaseSchemaTests` (replaces Android's `MigrationTest`).
- **`kolco24/Data/Records/*+GRDB.swift`** (13 files) — GRDB `FetchableRecord`/`PersistableRecord` conformances as **extensions** on the `Model/` types (hand-written `init(row:)` + `encode(to:)`, `databaseTableName` = Room table name), so `Model/` stays import-GRDB-free. JSON columns (`teams.members`, `marks.present`, `marks.presentDetails`) are coded here (analog of Room `TypeConverter`): `JSONEncoder(.sortedKeys)`, unknown keys ignored on decode, decode error → fallback (`[]` / `nil`) + log, never crash. `TeamMemberItem` uses `CodingKeys` with `number_in_team`.
- **`kolco24/Data/Stores/*.swift`** — 12 store **structs** (one per Android DAO — `struct`, not `protocol`: real stores over an in-memory DB serve both tests and the stage-3 repositories; no fakes). Kotlin `suspend` → `async throws` via GRDB `read`/`write`; Kotlin `Flow` → `ValueObservation.tracking{…}.values(in:)` (reactivity is wired now; stage 4 only subscribes). **SQL transcribed verbatim** rule: complex queries (team `startNumber` sort, `COALESCE(trustedTakenAt, takenAt)`, CASE upload-count aggregates) are ported as literal SQL strings — the goal is line-by-line checkability against the Kotlin DAO, not a pretty DSL. Shared DAO helper types (`UploadCounts`, `TrackScope`, `PhotoFrameRow`) live in `UploadTypes.swift`. Richest store is `MarkStore` (transactional `addMember` read-modify-write, `attachLocation`/`attachPhotos`, version-guarded `…IfUnchanged` family, photo/frame upload-drain aggregates); `CheckpointStore.replaceAllForRace` is **preserve-reveal** (snapshot revealed rows → wipe+insert server rows → re-apply plaintext, all in one transaction).

**Key-value stores** (only the two with a stage-2/3 consumer). Same idiom as `TrustedClock`: a pure core with injected `load`/`save` closures + a thin `fromUserDefaults` adapter over `UserDefaults.standard` (no separate suites — iOS has no Android backup-rules split; a stale anchor restored from backup is caught by `TrustedClock`'s monotonic-regression check).
- **`kolco24/Core/Stores/InstallId.swift`** — get-or-create UUID, key `install_id` (feeds stage-3 `X-Install-Id` header + `judge_scans.sourceInstallId`).
- **`kolco24/Core/Time/ClockAnchorStore.swift`** — key `anchor`, delimited format `"serverEpochMs|anchorElapsedMs|capturedWallMs|bootCount?"` 1:1 from Kotlin. Port trap: with `bootCount == nil` the string ends in `|`; parse via `components(separatedBy:)` and require exactly 4 components (Swift `split` drops the trailing empty segment).
- **`TrustedClock.makeDefault()`** (in `SystemClockProviders.swift`) — factory wiring the system providers to `ClockAnchorStore`'s `persist`/`persisted`. Actual app wiring lands in stages 3–4.

**Deferred** (explicitly out of stage 2, no code yet): `ThemePreference`/`TrackProfilePreference` (stages 8–9), `AdminTokenStore`/Keychain (stage 10), `RaceLeaseStore` (stage 9).

## Network & sync layer (`Net/` + `Data/Repositories/`)

Android-port stage 3 (see [`docs/plans/completed/20260709-android-port-stage3.md`](docs/plans/completed/20260709-android-port-stage3.md)) added the URLSession network slab + 4 sync repositories, ported from `kolco24_app_v2`'s `data/api/` + `data/*Repository.kt` with ~148 JVM tests mirrored into `kolco24Tests/Net/` and `kolco24Tests/Data/Repositories/`. **Not a 1:1 port** — URLSession has no OkHttp interceptor chain, so what Android splits across `AppSignatureInterceptor` + `ServerTimeInterceptor` collapses into one explicit pipeline inside `ApiClient`. The *behavioral* server contract (canonical strings, "data → then ETag" order, retry/RTT rules) is transcribed exactly — it's the spec locked by the mirrored tests. Consumers land later: stage 4 wires repositories to UI, stage 6 reuses the generic `post` pipeline for uploads, stage 9 substitutes a lease into the `isRacePinned` seam.

- **`kolco24/Net/ApiClient.swift`** — a `struct` whose deps are values + closures (the `TrustedClock` idiom), not injected services. One `send(...)` pipeline replaces both interceptors, in OkHttp-chain order: build `URLRequest` → sign with **6** `X-App-*` headers → `transport` with RTT timing → `ServerTimeSampler` → `await onServerTime?` → **403-retry-once** for GET/HEAD only when `ts` changed. Six headers: `X-App-Key-Id`, `X-App-Sig` (lower-hex HMAC of the `buildCanonical` string, stage-1 `Core/Api/Signing`), `X-App-Ts` (unix **seconds**, re-read on retry), `X-Install-Id`, **`X-App-Platform: ios`** (Android sends `android`), `X-App-Version` (`CFBundleShortVersionString`). `Authorization: Bearer` is added when `tokenProvider() != nil` but is **not** in the canonical string. `nowSeconds`/`onServerTime` are `async` closures with **inline `await`** (both are `TrustedClock`-actor calls) — the retry decision reads `nowSeconds()` *strictly after* `onServerTime` re-anchors, or 403 self-healing goes flaky (fire-and-forget `Task {}` would read the stale `ts`). Generic `post` serializes the body **once** to `Data` — the same bytes are hashed into the signature and sent — and **never retries** (403 is auth-vs-skew-ambiguous, replay unsafe); GET/empty body → `EMPTY_BODY_SHA256`. Endpoints are conditional GETs (`If-None-Match` verbatim with quotes): `fetchRaces`/`fetchTeams`/`fetchLegend`/`fetchMemberTags` (all ETag/304) + `fetchSync` (no ETag, lease manifest — stage 9 consumer). Paths carry a **trailing slash** (it's in the signed canonical path); `baseURL` is stored without one.
- **`kolco24/Core/Time/ServerTimeSampler.swift`** — pure port of `ServerTimeInterceptor.kt`: RFC-1123 `Date` header + `requestElapsedMs`/`responseElapsedMs` → `ServerTimeSample(serverEpochMs, anchorElapsedMs)?`. Server time is pinned to **RTT midpoint** (`requestElapsedMs + rtt/2`, overflow-safe form); missing/malformed `Date`, negative RTT, or RTT over the Kotlin limit → `nil` (no-op). `wallNow`/`bootNow` for `TrustedClock.onServerTime` are **not** the sampler's job — the factory wiring closure captures them from `SystemClockProviders`.
- **`kolco24/Net/ApiResults.swift`** — `FetchResult<T>` (`.success(data:etag:)`/`.notModified`/`.forbidden`/`.error(code:)`) and `PostResult<T>` (`.success`/`.badRequest`/`.unauthorized`/`.forbidden`/`.conflict`/`.rateLimited`/`.offline`/`.error(code:)`) — Kotlin `sealed interface` → Swift `enum`, **errors are values, never thrown**. Kotlin asymmetry preserved: a transport `URLError` folds to `.error(nil)` on GET but `.offline` on POST (offline mid-race is an expected upload state). `code == nil` = `URLError` or a parse error.
- **`kolco24/Net/Dto/*.swift`** (5 files) — `Codable` wire-only types; `Model/` mapping is done by the repositories. Pointwise snake_case `CodingKeys`; unknown keys ignored (incl. `sync.versions`); forward-compat defaults via `decodeIfPresent ?? default` (`total_cost`/`scoring_count` → 0, `tags` → `[]`, `TeamDto.start_number` optional, `CheckpointDto.cost/description/enc/color` all optional — `enc != nil` is the locked sentinel). Traps: `start_time`/`finish_time` are **ms** (`Int64`, 0 = none), `paid_people: Double`, legend tag key is `checkpoint_id`, `EncDto{iv, ct}` are base64.
- **`kolco24/Net/URLSessionTransport.swift`** — the transport closure seam's prod impl (`ApiClient.transport` = its bound `send`): `URLSessionConfiguration.ephemeral`, `urlCache = nil`, `.reloadIgnoringLocalCacheData` (the `Date` header — including on 304 — must be a live network value, it's what anchors the clock), single per-request timeout. Also the **two-client factory** `enum ApiClients` (port of `AppContainer.kt`): `makeCloud` (`Secrets.apiBaseURL`, 10 s, `onServerTime` → `TrustedClock`) and `makeLocal` (`Secrets.localAPIBaseURL`, 3 s for fast offline-fail off-Wi-Fi, **`onServerTime = nil`** — the LAN host never anchors trusted time). Both sign identically off the same `trustedClock.signingSeconds`. `makeDefaultPair()` builds the pair over a shared `TrustedClock.makeDefault()` + `InstallId.fromUserDefaults()`. App composition (full `AppContainer` analog) is stage 4.
- **`kolco24/Data/Repositories/*.swift`** (4 `struct`s over stage-2 stores) — shared **refresh flow**, a behaviorally exact port because it's the server contract: `(client, originKey)` picked by `SyncSource` → `syncMetaStore.getEtag` → conditional GET → on `.success`: **`deleteEtag` of the other origin (before the replace)** → `replaceAllForRace` → `upsert` new ETag (only if the server sent one). **Three separate transactions, "data → then ETag" order** is critical: a crash between them leaves fresh data with a stale/absent ETag (next refresh gets a harmless 200 and self-heals); the reverse order would pin a new ETag to old data forever. `RefreshResult` (`.updated`/`.notModified`/`.offline`/`.forbidden`/`.httpError(Int)`/`.skipped`) lives beside `RaceRepository`. **`isRacePinned: (Int) -> Bool` pin-guard seam** (currently always `false`; stage 9 substitutes a lease): checked before the network call **and re-checked after the 200** (source can flip mid-flight — a stale origin's response must not clobber the other's fresh rows). `sync_meta` keys: origin = base URL (cloud/LAN partition); resource = `"races"` / `"race/<id>/teams"` / `"race/<id>/legend"` / `"race/<id>/member_tags"`.
  - **`RaceRepository`** — the refresh-flow reference; **no pin-guard** (races are global, not race-scoped). `raceStore.replaceAll` (global).
  - **`TeamRepository`** — pin-guarded; `teamStore.replaceAllForRace` (one transaction over categories + teams). Maps `order` → `sortOrder`.
  - **`MemberTagsRepository`** — pin-guarded; plus a **synced marker** `sync_meta["race/<id>/member_tags/synced"] = "1"` written on a successful 200 that carries **no** server ETag — distinguishes "pool empty but synced" from "never synced". `hasBeenSynced`/`observeHasBeenSynced` check either the ETag resource or the marker (over `SyncMetaStore.observeEtagsExist`); both the ETag and marker of the other origin are cleared before the replace.
  - **`LegendRepository`** — pin-guarded; persists to **three** stores: `checkpointStore.replaceAllForRace` (**preserve-reveal** — an offline-revealed CP is not re-locked), `tagStore.replaceAllForRace`, `legendMetaStore.upsert`. Maps `CheckpointDto` (`enc != nil` → locked, `color ?? ""`). ## Scan / bind layer (`Nfc/` + `Location/` + `Audio/` + `App/ScanModel`)

Android-port stage 5 (see [`docs/plans/completed/20260709-android-port-stage5.md`](docs/plans/completed/20260709-android-port-stage5.md)) made the app useful on race day: NFC scan of the CP chip + wristbands → persist the take to the DB, a one-shot GPS fix at take-time (anti-fraud), sound/haptic feedback, and chip-to-member binding (`BindChipSheet`). **Not a 1:1 port of structure** — Android opens the scan overlay *from the chip itself* (permanent reader mode); iOS has no permanent reader mode (K24 chips aren't NDEF, so no background reads), so the overlay is always entered from the `MarksView` FAB and driven by **one long `NFCTagReaderSession`**. Deferred to later stages: upload of takes (stage 6 — the overlay-close hook is a no-op), photo marking (stage 7), GPS track (stage 8), chip write/provisioning via `writeRecord` 0xA2 (stage 10 — pure logic already tested), celebrations/skew banner (stage 11).

**Three new platform-adapter layers** join `Net/` as the app's only framework-bound slabs; each is the *sole* home of its import (grep-invariant, below), and none is unit-tested (adapters are exercised on-device — the pure seams they sit behind carry the tests):
- **`kolco24/Nfc/`** (`import CoreNFC` only) —
  - **`MiFareTransport.swift`** — `NfcTransport` (stage-1 `Core/Nfc/ChipRecord`) over `NFCMiFareTag.sendMiFareCommand`, port of the read side of Android's `MifareUltralightWriter`. The command sequence (FAST_READ 0x3A → 2×READ 0x30 fallback, K24 parse) stays in `Core/Nfc/readRecord`; this is just the callback→sync bridge. **Deadlock trap:** `readRecord` synchronously loops `transceive`, so each call blocks a `DispatchSemaphore` until CoreNFC delivers the `sendMiFareCommand` callback — that blocking `wait()` **must not** run on CoreNFC's callback (session delegate) queue, or it deadlocks the very serial queue the callback needs. The transport doesn't pick a queue; the **caller** guarantees it (`NfcChipScanner` runs `readRecord` on a dedicated background `readQueue`).
  - **`NfcChipScanner.swift`** — prod `ChipScanning` over `NFCTagReaderSession` (`final class`, state guarded by `NSLock`, delegate on a dedicated non-main queue). One long session per open overlay: `start()` opens the system sheet, `didDetect` → `connect` → UID (`normalizeNfcUid`) → clock sample **before** the read (§8) → blocking `readRecord` on `readQueue` → yield `TagReading` → `restartPolling()` for the next chip. `didInvalidateWithError`: `userCanceled` (including our own `invalidate()`) → finish the stream (host closes overlay); the **60 s iOS `sessionTimeout`** / read error + host says "window still alive" (`shouldRestart` closure → `ScanModel.shouldContinueScanning`) → silently begin a fresh session (the stream does **not** finish — a brief sheet flicker for the user). ~1.5 s same-UID debounce (kills audio spam; the reducer is idempotent anyway). `setStatus(_:)` (a `ChipScanning` protocol method the host pushes: "Приложите чип КП" / "КП 32 · чипы 2/4" / diagnostic) sets `session.alertMessage` and is re-applied after each read (`restartPolling` drops the prior string). One long always-continuous session for **both** scan and bind — no one-shot mode (bind mirrors Android's `DisposableEffect` reader hook that stays armed until `onDispose`, so re-tapping after `poolNotReady`/`notInPool` works). Async→sync deadlock is avoided exactly as `MiFareTransport`'s header warns.
- **`kolco24/Location/`** (`import CoreLocation` only) — **`CoreLocationProvider.swift`** — prod `CurrentLocationProvider` (`Core/Track/CurrentLocation`) over `CLLocationManager.requestLocation()` (`kCLLocationAccuracyBest`). `current(timeoutMs:)` races two child tasks in a `withTaskGroup` (delegate fix vs 8 s timeout); the loser is cancelled and its continuation resumed `nil` from a cancellation handler (else the group hangs). Maps `CLLocation → RawFix`: invalid `horizontalAccuracy` (< 0) → `Float.greatestFiniteMagnitude` (the pure `sanitizeFix` collapses it to `nil` back in `ScanModel`), altitude/vertical accuracy only when `verticalAccuracy > 0`. iOS `CLLocation` has no monotonic stamp like Android's `elapsedRealtimeNanos`, so it's synthesized from `mach_continuous_time()` minus the fix's wall-age — feeding both `isFixFresh` (≤10 s) and the `elapsedRealtimeAt` column consistently. Never throws: timeout / denial / error / stale → `nil` (take without coordinates; the server allows it). `requestWhenInUseAuthorization()` is the ahead-of-time permission hook (first overlay open).
- **`kolco24/Audio/`** (`import AVFoundation` + `import UIKit`-for-haptics only) — **`ScanFeedbackPlayer.swift`** + 4 WAV assets (`beep_ok3`, `beep_err`, `checkpoint_mark_completed`, `mark_added_mario`, copied from `kolco24_app_v2/app/src/main/res/raw/`). Prod `ScanFeedbackPlaying`, port of `data/ScanFeedbackPlayer.kt`: one `AVAudioPlayer` per clip (Android `SoundPool` analog; decoded in `init` but **no `prepareToPlay` there** — it implicitly activates the session and would duck the user's podcast/music from app launch), `AVAudioSession` `.playback` + `.mixWithOthers`/`.duckOthers` (the beep must cut through headphone music). **Duck lasts only for the clip**: the session is activated before playback and deactivated (`.notifyOthersOnDeactivation` restores the other app's volume) via a deferred work item at clip end + 0.3 s tail on a private serial `sessionQueue` — a new clip cancels/reschedules it, so the success-beep → fanfare (275 ms) pair doesn't bounce the session. `play(success/failure/neutral)` + `fanfare()`; haptics (`UIImpactFeedbackGenerator`/`UINotificationFeedbackGenerator`) are **best-effort** — the active NFC sheet may mute them, so sound is the primary channel. Best-effort by contract: any playback failure is swallowed, the scan flow never crashes.

**`Core/` additions (pure, tested):**
- **`Core/Marks/KpTake.swift`** — `makeKpTakeMark(...) -> Mark`, port of the *construction* half of `MarkRepository.startKpTake` (persist stays with the caller, `ScanModel`). One `distinctBy { numberInTeam }` pass over the buffer feeds both `present` (scoring truth) and `presentDetails` (upload snapshots); `complete = expectedCount > 0 && present.count >= expectedCount`; `method="nfc"`; `takenAt/updatedAt` = raw `sample.wallMs`, `trustedTakenAt/elapsedRealtimeAt/bootCount` from the same `TimeSample`. - **`Core/Scan/ChipScanning.swift`** — the scan seams (no Kotlin analog — Android's reader-mode wiring is smeared across `MainActivity`): value `TagReading { code: Data?; uid: String; sample: TimeSample }` (`code == nil` = valid non-K24 wristband read, **not** an error — §9) + `protocol ChipScanning: AnyObject` (`readings() -> AsyncStream<TagReading>`, `start()`/`stop()`; stream end = session ended not by the host) + `protocol ScanFeedbackPlaying` (`play(_:)`/`fanfare()`). `ScanModel` depends only on these, staying CoreNFC-/UIKit-free.

Mirror: `CurrentLocationTests`.

**`App/ScanModel.swift`** (`@Observable @MainActor`, `import Observation`/`Foundation` only) — the host reducer, porting *behavior* (not structure) of three Android spots: the `MainActivity.onScanTag` reducer (~1329–1468, the hardest part of the port), `ScanTakeState` (~475–483), and the `ScanScreen.kt` timer/fanfare/auto-close. Two mirrored states (§1): the pure `ScanSession` (UI: CP, present, window) + non-observable take-state for the DB (`markId`/`takeCheckpointId`/`expectedCount`/`buffer`/`takePresent`/`snapshots`/`takeLastScanAt`, all `@ObservationIgnored`). The window is timed off the monotonic `TagReading.sample.elapsedMs` (§8 — sampled by the scanner *before* the chip read). The §1–10 behavioral spec (locked by `ScanModelTests`, 13 cases):
- One `for await` over the `ChipScanning` stream serializes handling (§7 — replaces Android's `scanMutex`).
- Per reading: `legendRepository.unlock(raceId, code)` → fresh `checkpointsSnapshot` (only on `.revealed`/`.identityOnly`) → `classifyTag(code:uid:unlock:bindings:checkpointsById:)`; `bindings` = observed `[uid: numberInTeam]` scoped to roster slots (a stale chip of a removed member reads as `unboundChip`, never impersonates); empty-roster guard → `.badKp("команда не выбрана")`.
- `.kp` with expired window / `markId == nil` / different CP → **new take**: `makeKpTakeMark` → `markStore.upsert`; a repeat of the same CP within a live window only re-stamps `takeLastScanAt`. On expiry the buffer + snapshots are cleared before the new take.
- `.member`: after a dead window → **full take-state reset** before buffering (§4 — a member after a dead window opens a fresh session, isn't credited to the old take). The snapshot is built **before** idempotency checks; before CP → buffer (idempotent repeat exits early, window untouched), after CP → `present` guard + `markStore.addMember`. `takeLastScanAt = now` only for genuinely new members.
- `.unboundChip`/`.badKp` → diagnostic + failure feedback only; session and window untouched (§5).
- DB writes (`upsert`/`addMember`/`attachLocation`) run in **unstructured `Task`s** capturing the *stores* (not `self`), so closing the overlay never aborts an in-flight write (§6 — `applicationScope` analog; the take lives in the DB from the moment the CP chip is scanned).
- GPS is **fire-and-forget once per new take** (`attachLocationForNewTake`): a captured `Task` awaits `locationProvider.current()`, `sanitizeFix`, then `markStore.attachLocation` — `nil` fix = no-op, slow GPS never blocks the window.
- Feedback: `feedbackFor(event)`; on the incomplete→complete edge, a normal success then `fanfare()` after `fanfareDelayMs` (275). Timer: 250 ms tick over the injected `elapsedNowMs`; on expiry → finalize + `closeRequested`; on `isComplete` → hold "Готово!" `successHoldMs` (3300) → `closeRequested`. All timings are injectable for controlled-time tests; `shouldContinueScanning()` is the scanner's 60 s restart gate.

## Upload layer (`Core/Upload/` + `Net/Dto/MarkUpload` + `Data/Repositories/MarkUploadRepository` + `App/UploadModel`)

Android-port stage 6 (see [`docs/plans/completed/20260710-android-port-stage6.md`](docs/plans/completed/20260710-android-port-stage6.md)) made the app race-usable (the MVP border): takes accumulated in the DB with `uploadedLocal`/`uploadedCloud = 0` are now idempotently drained in batches to **both** targets — cloud (HTTPS, "Интернет") and the LAN finish server (local, "Финиш") — with independent flags, self-heal on offline/error, plus a "Загрузка данных" screen with counters + force-flush. **Not a 1:1 port of structure** — Kotlin's `Mutex.tryLock()` becomes an `actor` + `inFlight` flag, the `MainActivity` trigger soup becomes `AppModel` methods, and Android's per-data-type `UploadScreen` sections collapse to one ("Отметки") since only marks drain now. Deferred: photo-frame drain (stage 7 — `combineOutcome` is already the groundwork), track drain (stage 8), judge_scans (stage 10), LAN lease (stage 9), BGTaskScheduler background flush (out of MVP).

- **`kolco24/Core/Upload/UploadModels.swift`** (pure Foundation, no framework imports) — mirror of `data/track/TrackModels.kt` L79–98 (upload-part): `enum UploadTarget` (`local`/`cloud`), `enum UploadResultKind` (`ok`/`offline`/`error`), `struct TargetUploadOutcome { kind, atWallMs }` (`Equatable`; wall-clock stamp for the "N мин назад" line), and `combineOutcome(_ metadata:_ frame:) -> UploadResultKind?` — deterministic precedence **`error > offline > ok > nil`** (a frame `ok` must never mask a metadata `error`/`offline`; `nil` only when neither cycle tried anything). `combineOutcome` is stage-7 groundwork (metadata + photo frames); stage 6 calls it trivially as `combineOutcome(meta, nil) == meta`.
- **`kolco24/Net/Dto/MarkUpload.swift`** — the wire types for `POST /app/race/<raceId>/marks/`, mirror of `data/api/dto/MarkDtos.kt`. `MarkUploadRequest { team_id, source_install_id, marks }` (`source_install_id` = the same UUID as the `X-Install-Id` header, but **duplicated into the signed body** — the marks contract needs it in-body so the server can dedup two phones of one team), `MarkDto`, `PresentMemberDto`, `TakeLocationDto` (all `Encodable`); `MarkUploadResponse { accepted: [String] }` (`Decodable`). Three key traps:
  - **`present[]` merge** (`MarkDto(from mark:)`) — the array is built over `Mark.present` (the **scoring truth**), enriched from `presentDetails.associateBy(numberInTeam)`; a slot with no snapshot emits the sentinel `PresentMemberDto(nfc_uid: nil, code: nil, number: 0, number_in_team: num)` (**not** a real member number). So a legacy row (`presentDetails == nil`) still uploads every member as a sentinel, and a partially-snapshotted row enriches only the slots that have a snapshot — no member is ever dropped.
  - **Nullable-encoding rule** (reproduces kotlinx.serialization `encodeDefaults = false`, `explicitNulls = true`) — no-default scalars (`trusted_ms`, `elapsed_at`, `boot_count`, `present[].nfc_uid`/`code`, every nullable field of `TakeLocationDto`) are written as an explicit JSON `null` via manual `encode(...)`; but `location` carries a Kotlin default `= null` and is **omitted entirely** (no key) when there's no fix — the one field using `encodeIfPresent`. Hence the hand-written `encode(to:)` on each DTO instead of the synthesized one.
  - **Rename traps** (`CodingKeys` are **not** the plain snake_case of the Swift field names): `cp_nfc_uid ← cpUid`, `cp_code ← cpCode`, plus `wall_ms ← takenAt`, `trusted_ms ← trustedTakenAt`, `elapsed_at ← elapsedRealtimeAt`. `location` (nested `TakeLocationDto`, built from the 7 `loc*` fields) is `nil` when `locLat == nil`.
  - `ApiClient.uploadMarks(raceId:teamId:sourceInstallId:marks:) async -> PostResult<MarkUploadResponse>` reuses the generic `post` pipeline (body serialized once — the same bytes are signed and sent; **never retries** — 403 is auth-vs-skew-ambiguous, replay-unsafe) on the trailing-slash path `/app/race/<raceId>/marks/`.
- **`kolco24/Data/Repositories/MarkUploadRepository.swift`** (`import GRDB` — under `Data/`) — the drain engine, port of `data/MarkRepository.kt` L280–407 (upload-part).
  - **`actor` + `inFlight` tryLock analog** — actor isolation + a `private var inFlight` flag replace Kotlin's `Mutex.tryLock()`: overlapping triggers silently no-op (the loser returns, the next trigger catches up). Both entry points (`uploadPending(raceId:teamId:)`, `uploadAllPending()` over `pendingUploadScopes()`) guard on it.
  - **Generic `drainUploadLoop<Row>(fetch:id:upload:mark:) async -> UploadResultKind?`** — Kotlin `uploadLoop` (L390–407) semantics **1:1**: empty first fetch (nothing to send) → `nil` (no attempt); drained to empty after ≥1 mark → `.ok`; non-`success` response → its `uploadResultKind`; **`accepted ∩ batch` empty → `.error`** (no-progress anti-loop guard — a strange response can't mark a row it didn't accept, and a marked row strictly shrinks the pending set → termination guaranteed); DB errors (`fetch`/`mark` throw) → `.error` + log (the engine never crashes the process — the stage-2 "decode error → fallback + log" convention). `upload` returns the already-extracted `accepted` (`PostResult<[String]>`, via a private `PostResult.mapSuccess`), not a concrete response DTO — so stages 8/10 reuse the loop for track/judge with their own `*UploadResponse`.
  - **`flushScope`: Local then Cloud**, each cycle independent (one target failing doesn't block the other), with **GPS-aware version-guarded marking**: per accepted row, `locLat != nil` → `markUploaded*IfUnchanged(id:updatedAt:)`, else `…IfUnchangedAndNoLocation`. The two guards are **orthogonal**: `addMember` bumps `updatedAt` (row re-uploads with fresh `present`) but doesn't touch `locLat`; `attachLocation` sets `locLat` (row re-uploads with the coordinate) but doesn't bump `updatedAt` — so a late GPS fix or a late member is never silently dropped by the mark. Reuses the stage-2 `MarkStore` drain API verbatim (`unuploadedLocal/Cloud`, the version-guarded `markUploaded*` family, `pendingUploadScopes()`, `TrackScope` as the outcomes dict key).
  - **Outcomes** — `outcomes: [TrackScope: [UploadTarget: TargetUploadOutcome]]` + `nonisolated let outcomeUpdates: AsyncStream<…>` (`.bufferingNewest(1)`, equal snapshots deduped by hand — the `TrustedClock.statusUpdates` idiom); `wallNow: () -> Int64` is injected for controlled-time tests. `uploadResultKind(PostResult)` maps `.offline → .offline`, everything else → `.error` (a clean `.success` never reaches it — the loop handles it).
  - Constants `uploadBatch = 500` (`UPLOAD_BATCH`), `AppModel.uploadRetryIntervalMs = 300_000` (`UPLOAD_RETRY_INTERVAL_MS`) — names as in Kotlin.

## Map layer (`Core/Map/` + `Data/MBTilesReader` + `Map/` + `App/MapModel` + `MapTabView` + migration v2)

The **iOS-only** 4th tab «Карта» (no Android mirror — see [`docs/plans/completed/20260717-map-tab.md`](docs/plans/completed/20260717-map-tab.md)): the team's live GPS track + taken КП over a raster **offline basemap** (one MBTiles file per race, sliced server-side from the existing XYZ tiles and served as static content). The user downloads it with an explicit button before the start (over Wi-Fi); afterwards the map works fully offline. Without the file the tab **degrades to the online Apple basemap** carrying the same track + КП — for free, since MapKit renders Apple tiles whenever no overlay is added. No new dependency: GRDB (already in the project) reads the `.mbtiles` sqlite file, MapKit renders. **KP coordinates do not come from the server** — each pin is the take's own GPS fix (`Mark.locLat`/`locLon`, written by `attachLocation` at scan time); a take with **no fix is not shown** (deliberate — the server has no КП coordinates to send).

- **`kolco24/Core/Map/MBTiles.swift`** (pure Foundation — the sole home of the format math) — `tmsRow(z:y:) = (1 << z) − 1 − y`, the **TMS y-flip trap**: MBTiles stores rows in the TMS scheme (`tile_row = 2^z − 1 − y`, y = XYZ/«google»), so the flip lives in **one place** and `MBTilesReader` calls it on every tile read. Plus `struct MBTilesMetadata` (`bounds` W,S,E,N / `center` / `minZoom` / `maxZoom`, all optional) + `parseMBTilesMetadata([String:String])` — **never-throw** (missing/garbage → nil fields, the stage-2 «decode error → fallback» convention): `bounds` requires **exactly** 4 dot-decimal numbers (`Double(_:)` locale-independent), zooms are ints. Mirror: `MBTilesTests` (y-flip boundaries `z=0`/max-y; metadata full/partial/garbage/empty).
- **`kolco24/Data/MBTilesReader.swift`** (`import GRDB` — under `Data/`, which now also holds this reader) — a read-only `DatabaseQueue` (`Configuration.readonly = true`) over the `.mbtiles`. `tileData(z:x:y:)` takes **XYZ** input, applies `tmsRow` from Core, `SELECT tile_data … WHERE zoom_level/tile_column/tile_row` — missing tile / read error → `nil` + log (never-throw); `metadata()` folds `SELECT name, value FROM metadata` → `parseMBTilesMetadata` (broken table → `nil`). Only `init(path:)` throws (file absent / not sqlite). **Concurrency note (locked by comment):** `MKTileOverlay.loadTile` is called concurrently from MapKit background queues; one `DatabaseQueue` serializes reads through a single connection — enough for pan/zoom raster tiles, and a read-only file can't open a WAL `DatabasePool` anyway. Mirror: `MBTilesReaderTests` (a tiny `.mbtiles` built in-test via GRDB with TMS-row inserts → tile read y-flip correct, missing → `nil`, metadata parses, absent file → `init` throws).
- **`kolco24/Map/`** — a new platform folder, the **only `import MapKit` home** (grep-invariant; precedent `Photo/CameraPreviewView`):
  - **`MBTilesOverlay.swift`** (`import MapKit`) — `MKTileOverlay` subclass: `canReplaceMapContent = true` (Apple tiles never load), `tileSize` 256, `minimumZ`/`maximumZ` from `MBTilesMetadata` (nil → defaults 0…19). Tiles arrive via an injected `@Sendable (Int,Int,Int) -> Data?` closure (prod = `MBTilesReader.tileData`), so **GRDB does not reach `Map/`** — `MBTilesReader` under `Data/` stays the sole `import GRDB`. Missing tile → empty transparent `Data()` (not an error, so MapKit logs no load failure).
  - **`TrackMapView.swift`** (`import MapKit` + `import SwiftUI`-for-tokens) — `UIViewRepresentable` over `MKMapView`. Inputs: `trackPath` (`Double` pairs), `pins: [MapMarkPin]`, an optional `MapOverlayDescriptor` (metadata + tile closure); `nil` overlay → the Apple basemap. **Coordinate-type boundary:** `CLLocationCoordinate2D` appears **only here** — the conversion from the `Double` lat/lon pairs `MapModel` prepares (otherwise `App/MapModel` would need `import CoreLocation`, breaking the invariant). Polyline = `MKPolyline` in `kolcoOrange` ~3 pt, **full replace** on data change (no incremental append); КП = a custom `MKAnnotationView` (a simplified `CPBadge` without stripes — `brandRed` circle, white `Font.mono` number) with a callout «КП N · M баллов · HH:mm» (`PluralRu.pointsLabel`). Camera: with an overlay → region from `bounds` + `cameraBoundary` bbox + `cameraZoomRange` from the zooms; without → a one-shot fit to the track/pins. `showsUserLocation = true` (the When-In-Use grant already comes from the scanner; no permission → the dot just doesn't appear). Device-only, no unit tests.
  - **`MapFileStorage.swift`** — the on-disk adapter + downloader, but **Foundation-only** (`URLSession` + `FileManager`; unlike the two MapKit files, no `import MapKit`). **One-file offline delivery:** `rootURL/maps/<raceId>.mbtiles` (beside `kolco24.db` in Application Support, the `PhotoStorage` precedent) — the file's presence **is** the «downloaded» flag, no separate prefs bit. `download(from:raceId:onProgress:)` uses a `URLSessionDownloadTask` + delegate (`didWriteData` → progress; `totalBytesExpected ≤ 0` → indeterminate 0) — **not** `URLSession.bytes` byte-iteration, which for 20–60 MB is O(n) awaits and painfully slow. On finish: HTTP ≠ 2xx → `MapDownloadError.httpError`, else **atomic** `replaceItemAt`/`moveItem` into place (the file appears only whole); cancel/error → the temp is removed, nothing lands. **No resume** — a broken/cancelled download restarts from scratch (the size is known, resume-data isn't used). Disk half tested in a temp dir (`MapFileStorageTests`: `fileURL` scheme, `exists`/`fileSize`/`delete`, atomic overwrite on re-download); the network download is device/integration only.
- **`kolco24/App/MapModel.swift`** (`@Observable @MainActor`, imports only `Observation`/`Foundation`) — the tab reducer. Subscribes to `trackStore.observeForTeam(teamId:raceId:)` + `markStore.observeForTeam(_:)` + `checkpointStore.observeCheckpointsForRace(_:)` with the same **`rebind(teamId:raceId:)` stale-guard** as the other per-tab models (cancel + clear arrays before restart; a genuine team change also cancels an active download and recomputes availability for the new race). **Derived (all `Double`, no `CLLocationCoordinate2D`):** `trackPath = sortedTrackPoints(filterPoints(trackPoints))` (the same `trackUsable` calc as `TeamModel`); `pins: [MapMarkPin]` = marks with `locLat != nil`, number from `checkpointsById[mark.checkpointId]` (fallback `mark.checkpointNumber`), cost `checkpointCosts[id] ?? mark.cost` (the `MarksModel.costOf` idiom — a locked CP without a price falls back to the take snapshot), time `trustedTakenAt ?? takenAt`. **State machine `MapAvailability`:** `noMapForRace` (race `mapUrl == nil`) / `notDownloaded` / `downloading(progress:)` / `ready(path:)` / `failed(message:)` (→ toast, CTA back to `notDownloaded`). `refreshAvailability()` recomputes from the filesystem + the race's `mapUrl` (read via **`RaceStore.getById`**, added this stage) — called from `MapTabView`'s `.task`/`onAppear` because tabs in a `TabView` live permanently and deleting the map in Settings otherwise wouldn't reach the already-built model (the file-as-flag isn't observable). `downloadMap()` guards re-entry, streams progress, success → `ready`, error → `failed` + toast; `cancelDownload()` cancels the `Task` (storage sweeps the temp) → `notDownloaded`. The download captures the graph **closure** (not `self`), the §6 idiom. `AppModel.makeMapModel() -> MapModel?` mints it for the current `raceId`/`teamId` (nil without a team — the `makeUploadModel` idiom). Tests: `MapModelTests` over `AppEnvironment.inMemory` + fake closures (state transitions, download success/error/cancel, `ready` on an existing file, `refreshAvailability` rollback after delete, derived filtering / no-fix pin drop / price fallback, stale-guard on team change) + a `RaceStore.getById` store test.
- **`mapUrl` through the data slab (migration v2 — the FIRST iOS-only divergence from Room v5).** The iOS DB is **no longer born at v5**: `AppDatabase.migrator` now has a second migration **`"v2"` = `ALTER TABLE races ADD COLUMN mapUrl TEXT`** (existing installs migrate on start; `makeInMemory()` also runs up to v2). `Race` gains `mapUrl: String?` (default `nil` in `init` — existing constructions untouched); `Race+GRDB` codes the column. `RaceDto` adds `let mapUrl: String?` + `case mapUrl = "map_url"` — the synthesized `Codable` yields `nil` on a missing key/`null` (**forward-compatible**: the client works against a server with no field), mapped in `RaceRepository.toRace()`. **`map_url` must be HTTPS** — ATS in `Info.plist` allows cleartext only on the local network, so an HTTP static host would be blocked; the tiles aren't secret, so the URL carries **no HMAC signature** (the download-task isn't signed). Locked by `AppDatabaseSchemaTests` (the `races` column inventory now includes `mapUrl`, `migrationRunsOnEmptyDatabase` asserts `["v1","v2"]`, `migrationV1ToV2AddsMapUrlAndPreservesRows` migrates a non-empty `races` up to `"v1"` then to the end and checks rows survive with `mapUrl == nil` — `makeInMemory()` won't do, it jumps straight to v2).
- **UI wiring.** `MapTabView.swift` (root `kolco24/`, design tokens): no team → `EmptyStates` `onChooseTeam`; with a team → `TrackMapView` full-screen + a state overlay (`noMapForRace` → an unobtrusive «Оффлайн-карта для этой гонки недоступна» line; `notDownloaded` → a «Скачать карту гонки» CTA card; `downloading` → a `ProgressView` card with percent + cancel; `ready` → the clean map; `failed` → a toast), with `model.refreshAvailability()` in `.task`/`onAppear`. `ContentView` gains the 4th tab «Карта» icon **`mappin.and.ellipse`** (not `map.fill` — taken by «Легенда»), same tab-switch haptic, `makeMapModel()`. `SettingsView`'s «Данные» section gets a «Удалить карту гонки» row (size subtitle via `SettingsModel.mapFileSizeLabel`, disabled without a file) → a destructive `confirmationDialog` calling `SettingsModel.deleteRaceMap()` (unstructured `Task` capturing the graph closure, §6). Tests: `SettingsModelTests` (label present/absent, `deleteRaceMap` fires the closure); UI views have no unit tests.
